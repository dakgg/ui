# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Unity UPM package — `com.dakgg.ui`, display name "ViewSystem UI Framework". A
stack-based view/UI manager for uGUI with focus & input control, layered
canvases, back-button handling, and Addressables-driven loading.

- **Assembly**: `Dakgg.UI` (runtime), `Dakgg.UI.Editor` (editor-only).
- **Namespace**: `ViewSystem` (note: not `Dakgg.*`). Some utility classes
  (`BundleUtility`, `UIAnimation`, `ExecutionOrder`) are in the global namespace.
- **Layout**: `Runtime/Core/` (view system), `Runtime/Utils/` (loading,
  animation, visibility helpers), `Editor/`.

This package lives at `client/Packages/ui` inside a larger Unity project
(`client/`), which is its own git repo. **This package directory (`Packages/ui`)
is its own git repository** — commits here are separate from the Unity project.

## Building / testing

There is no CLI build or test setup. Code is compiled by **opening the parent
Unity project (`client/`) in the Unity Editor** (Unity 2021.3+). The `.csproj`
files under `client/` are Unity-generated; don't hand-edit them. There are no
unit tests in this package.

`.meta` files are required for Unity packages and are generated by the Editor.
If you add a new asset/script, the matching `.meta` must be committed.

## Architecture

The whole system funnels through one singleton, **`ViewManager`** (MonoBehaviour,
`Core/ViewManager.cs`). `ViewManager.Instance` is set in `Awake`; most public API
is `static` and assumes the instance exists in the scene.

### Views and the layer stack

- **`ViewBase`** (`Core/ViewBase.cs`) is the base for all stacked views.
  Subclasses `ModalView` and `PageView` differ only in their `GetViewProperty()`
  override (`HideBehindViews`, `IsGlobal`, `LayerType`). `CameraPageView<T>` is a
  `PageView` that spawns/destroys a companion `IViewCamera` with the view.
- **`ViewLayer`** (`Core/ViewLayer.cs`) holds an ordered stack of active views.
  Two layers exist: `Default` and `Special` (created in `ViewManager.Awake`).
  Each layer owns a canvas-`sortingOrder` band of `MAX_COUNT_PER_LAYER` (100);
  views within a layer are spaced by `m_ViewGap` (10).
- **`OverlayView`** (`Core/OverlayView.cs`) is a *separate*, non-stacked static
  view type (e.g. tooltips). Managed via `AddOrReplaceOverlay`/`GetOverlay`, not
  the layer stack.

### Open/close flow

- Open a view with **`ViewRequest.Open<TView>(openAction, instant)`** (or the
  string-keyed overload). Returns a `ViewRequest` that is both a
  `CustomYieldInstruction` (yield in coroutines) **and** awaitable
  (`await ViewRequest.Open<T>()`).
- Close/return results via extension methods in `Core/ViewOperations.cs`:
  `view.Complete()` (cancel/close), `view.Success()` /
  `view.Success(result)`. For results, open with `ViewRequest<T>` and read
  `request.Result` / `TryGetResult`.
- The awaiter infrastructure (`IAwaiter`, `ViewRequestBase`) **enforces the Unity
  main thread** — `ViewManager.OnContinuation` throws if continued off-thread.

### Loading & naming

- Views are loaded via **Addressables**. The Addressables key is the type name
  plus the `@View` suffix (e.g. `LoginView` → `LoginView@View`), computed by
  `ViewNames.GetName`. Override the base path with the
  `[ViewLoad("path")]` attribute on the view class.
- `BundleUtility` (`Utils/BundleUtility.cs`) wraps all Addressables
  load/instantiate/release calls. Instantiated GameObjects auto-release their
  handle via a hidden `BundleReleaseTracker` on destroy.
- `ViewRootCanvas` (`Core/ViewRootCanvas.cs`) provides two parent canvases:
  a per-scene root and a global (`DontDestroyOnLoad`) root for `IsGlobal` views.
  Both are instantiated from `Resources.Load("ViewRootCanvas")`. Setting its
  `m_RootViewName` auto-opens a root view on `Start`.

### Focus, visibility, input — the Refresh cascade

`ViewManager.Refresh()` → `ViewLayer.Refresh()` walks every layer top-down and
recomputes, per view: canvas visibility (a view with `HideBehindViews` hides
everything beneath it), the single focused view, and per-view input
(`SetInputAllowedInternal` toggles `CanvasGroup.blocksRaycasts`). This runs
automatically on every add/remove. Per-view input can also be locked manually
with `view.SetInputEnable(bool)` (ref-counted).

### Lifecycle callbacks (marker interfaces)

A view opts into lifecycle events by *implementing* interfaces from
`Core/ViewCallbackInterfaces.cs` — there are no virtual `OnX` methods to
override. Key ones: `IViewAddedListener`, `IViewRemovedListener`,
`IViewInitialFocusListener` (Start replacement), `IViewFocusListener`,
`IViewBackButtonListener` (else the view auto-closes on back),
`IViewUpdateListener`, `IViewVisibleListener`. `IViewRoot` marks a view that
`CloseAll` will not pop.

### Back button

`ViewManager.Update` maps `KeyCode.Escape` to the focused view's back handler
(disable globally via `ViewManager.EscapeControl`). It also detects screen
orientation changes and fires `ViewManager.OnOrientationChange` (one frame
delayed so safe-area updates settle).

### Animation & visibility helpers

- `UIAnimation` (`Utils/UIAnimation.cs`) samples **legacy** `AnimationClip`s
  manually each frame (so it ignores `Time.timeScale` when realtime) and
  dispatches animation events by reflection to components on the GameObject.
  `AnimationSet` (in `ViewBase.cs`) pairs fold/unfold clips for open/close.
- `UIVisiblity` (`Utils/UIVisiblity.cs`) drives `IViewUpdateListener` /
  `IViewVisibleListener` based on whether the parent canvas is enabled — i.e.
  "update only while visible".

## Gotchas

- `README.md` is partly stale: it names the base class `UIBase`, but the actual
  base class is `ViewBase`. Trust the source.
- `ViewManager` static API throws if `Instance` is null — a `ViewManager` must
  exist in the loaded scene before opening any view.
- Open one view type at a time: `m_AvailableViews` is keyed by view name, so the
  same view type cannot be active twice concurrently (`TrackView` throws).
